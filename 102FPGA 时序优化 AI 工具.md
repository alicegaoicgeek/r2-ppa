# FPGA 时序优化 AI 工具

> 整合多 AI 分析后的工程级实施方案。每一步都给出**输入、动作、输出、决策、防护**五要素。

---

## 〇、设计原则（先立规矩）

| 原则 | 含义 | 为什么重要 |
|------|------|----------|
| **廉价先行** | 静态分析先跑，PnR 后跑 | 一次 PnR 几小时，一次静态分析几分钟 |
| **AI 提议，工具裁判** | AI 不能给指标，只能改方案 | 防止 AI 既当裁判又当选手 |
| **RTL 必过 LEC** | 任何 RTL 改写不过 LEC 不进循环 | RTL 改错可能让芯片报废 |
| **失败比成功值钱** | 失败案例必须回写 | 每次失败省下下次几小时 PnR |
| **越界要审计** | AI 跨边界的动作单独记日志 | 保留可控性 |

---

## 一、五阶段工作流总览

```
阶段              耗时       谁跑          产物              安全网
────              ────       ───           ────              ────
0. 基线建立       1×PnR      Vivado        当前 Fmax/WNS    无需
1. 静态预筛       分钟级     AI + LEC      安全改写池        LEC
2. 工具层搜索     N×PnR并行  AutoResearch  最优参数组合      Vivado
3. 越界探索       分钟+PnR   Agent         新假设            LEC + audit
4. 知识沉淀       秒级       自动          知识库更新        无需
```

---

## 二、阶段 0：基线建立

### 输入
- RTL 源码（SystemVerilog）
- 约束文件（XDC）
- testbench
- 目标频率（如 360 MHz）
- 资源/功耗预算
- 边界声明：哪些文件禁止动、哪些模块禁止改语义

### 动作
```bash
1. 完整跑一次综合 + 布局 + 布线
2. 解析时序报告：
   - 当前 Fmax / WNS
   - Top-N 关键路径
   - 每条路径的瓶颈分类（读改写、深加法、扇出、互连）
3. 跑 testbench → 拿到功能基准
```

### 输出
- `baseline.dcp`（PnR checkpoint）
- `baseline_timing.rpt`（时序报告）
- `baseline_paths.json`（结构化的关键路径清单）
- `baseline_funccheck.log`（功能基准）

### 决策点
| 条件 | 动作 |
|------|------|
| 当前 Fmax ≥ 目标 | 结束（无需优化） |
| 当前 Fmax < 目标 | 进入阶段 1 |

## 三、阶段 1：静态预筛（廉价大杠杆）

### 输入
- 阶段 0 的输出
- **历史知识库**：
  - 瓶颈模式目录
  - 策略-频率映射表
  - 避坑库

### 动作
```
1.1 路径分类
    AI 把每条关键路径映射到已知模式
    "这是读改写环路 / 这是深加法链 / 这是高扇出..."

1.2 候选生成（双轨）
    
    A 轨：参数候选（不改 RTL）
    └─ 查策略-频率表，列出当前频率下 Top-K 推荐策略
       例：360MHz 目标 → ExtraTimingOpt + AggressiveExplore
    
    B 轨：RTL 改写候选
    └─ 扫 RTL 找代数等价改写：
       - 共享比较器
       - 公共子表达式
       - "裁剪后判零"模式
       - 冗余加法
    └─ 每个候选附带：
       - patch diff
       - 自验证（枚举/代数推导）
       - 预期节省（ps，置信度）
       - 风险等级（低/中/高）

1.3 LEC 闸门（仅 B 轨经过）
    对每个 RTL 候选：
    └─ 跑 Vivado LEC / Yosys eqy
    └─ 通过 → 安全改写池
    └─ 失败 → 写入"AI 推理盲区"库（高价值！）

1.4 高风险改写人审分流
    动了 reset/CDC/状态机/协议的 → 强制人审
    纯组合逻辑化简 → LEC 通过即可
```

### 输出
- `safe_rewrite_pool/`（已过 LEC 的 RTL patch）
- `param_candidates.json`（参数组合候选）
- `lec_failures.log`（AI 自以为对但其实错的案例）
- `human_review_queue/`（高风险改写，等人审）

### 决策点
| 条件 | 动作 |
|------|------|
| 安全改写池非空 | 应用所有 patch，跑一次 PnR 确认基线提升 |
| 安全改写池为空 | 直接进阶段 2 |
| 人审队列非空 | 通知工程师，可选择跳过或等待 |

### 防护
- LEC 是硬门槛，不允许任何 RTL 改写绕过
- 人审改写不影响主流程推进，并行处理

## 四、阶段 2：工具层搜索（AutoResearch 主循环）

### 输入
- 阶段 1 的最佳基线（可能已应用 RTL 改写）
- 参数候选清单
- 计算资源（N 个并行 license）

### 动作
```
2.1 候选编排
    把候选按"预期收益 × 失败概率"排序
    每轮取 N 个并行跑（N = 可用 license 数）

2.2 单候选执行（每个并行任务）
    a. 创建分支 checkpoint
    b. 应用候选（修改 XDC 或 Tcl）
    c. 跑 incremental PnR（复用前阶段缓存）
    d. 解析时序报告 → WNS
    e. 跑 testbench 回归（应用了 RTL 改写时）
    f. 决策：
       WNS 改善 ≥ ε → keep（更新 best）
       WNS 不变 → keep 但标记"无效"
       WNS 退化 → discard，原因写回知识库

2.3 收敛判定
    每完成一轮（N 个候选）后检查：
    达到目标 → 结束
    连续 M 轮无改善 → 触发阶段 3
    候选池空 → 触发阶段 3

2.4 增量缓存策略
    综合后 / 布局后 / 布线后分别 checkpoint
    改后端选项时复用前端结果（省 50-70% 时间）
```

### 输出
- `best.dcp`（当前最优 PnR）
- `iteration_log.json`（每轮：候选、WNS、keep/discard、原因）
- `param_failures.log`（失败的参数组合）

### 决策点
| 条件 | 动作 |
|------|------|
| Fmax ≥ 目标 | 进入阶段 4 收尾 |
| 连续 M 轮无改善 | 进入阶段 3 越界 |
| 资源耗尽 | 进入阶段 4，输出当前最优 + 缺口报告 |

### 防护
- WNS 永远来自 Vivado，AI 不许估算
- 每个 keep 必须 testbench 通过
- 失败原因结构化记录（不只是"变差"，要分析为什么）

## 五、阶段 3：越界探索（触发式 Agent）

### 触发条件
阶段 2 撞天花板（连续 M 轮无改善），且未达目标。

### 动作（两类越界，分开处理）

#### 越界类型 A：跨抽象层（改 RTL）
```
A.1 重读关键路径 RTL
    带着"工具层已穷尽"的上下文重新推理
    重点找阶段 1 错过的：
    - 跨多个 always 块的等价关系
    - 与 testbench 行为相关的不变量
    - 与综合工具映射习惯相关的优化

A.2 候选 → LEC → 安全池
    严格走阶段 1 同样的闸门
    没有捷径

A.3 回到阶段 2 验证
    把新 patch 加入候选池继续跑
```

#### 越界类型 B：跨参数上限（改硬编码）
```
B.1 读迭代日志
    判断曲线形态：
    - 单调下降但慢 → 不调上限
    - 游荡未收敛 → 可能调上限
    - 达到上限时还在改善 → 应该调上限

B.2 调整方案
    例：phys_opt 迭代 5 → 10
    例：placement seed 数量 1 → 4

B.3 audit log
    每次跨上限必须记录：
    - 改了什么参数
    - 依据是什么（哪段日志）
    - 改后效果

B.4 回到阶段 2
    用新上限继续搜索
```

### 输出
- 新的 RTL patch（走 LEC 后入安全池）
- 新的参数上限配置
- `boundary_audit.log`（越界审计）

### 防护
- 类型 A 必过 LEC（同阶段 1）
- 类型 B 必入 audit log
- 越界次数有上限（防止无限发散）

---

## 六、阶段 4：知识沉淀（每轮自动）

### 输入
- 阶段 0~3 的所有日志、决策、改动

### 动作
```
4.1 抽象成功模式
    本次有效的优化 → 提炼为"模式名 + 适用条件"
    例："共享比较器" + "适用于裁剪后判零结构"
    入瓶颈模式目录

4.2 沉淀失败案例
    工具选项失败 → 入策略-频率反向表
    "在 360MHz 目标下，扇出复制无效，原因：挤散布局"
    
    RTL 改写失败 → 写回源码注释
    "//-- 此处试过 X 改写，LEC 失败/PnR 退化，原因：Y"

4.3 LEC 失败特别归档
    "AI 推理盲区"是最稀缺的训练数据
    单独存一个数据集，用于改进未来 AI 提议质量

4.4 频率敏感性标注
    所有沉淀的策略推荐必须带频率标签
    避免脱离上下文复用
```

### 输出
- 知识库更新（git commit，可追溯）
- 项目报告（含"本次新增 X 个模式"）

## 七、关键工程细节

### 7.1 并行策略（应对单次 PnR 成本）

```
每轮的 N 个候选并行跑：
- 不同布局策略 → 不同机器，互不干扰
- 不同 RTL patch → 不同 working dir
- 共享综合阶段 checkpoint，从布局开始 fork

实测：N=4 的并行 ≈ 串行 3.5 倍速度
```

### 7.2 增量编译

```
分级 checkpoint:
  post_synth.dcp    ← 改约束/布局策略时复用
  post_place.dcp    ← 改 phys_opt 参数时复用
  post_route.dcp    ← 改时序分析参数时复用

只改 phys_opt 不需要重跑综合，能省 60% 时间
```

### 7.3 testbench 加速

```
分级回归：
  smoke test       ← 每个 keep 必跑（5 分钟）
  full regression  ← 每轮结束跑一次（小时级）
  formal check     ← 入 main 前跑（小时级）

绝大多数错误能在 smoke test 抓住
```

### 7.4 知识库结构

```
knowledge_base/
├── bottleneck_patterns/        # 瓶颈模式目录
│   ├── read_modify_write.md
│   ├── shared_comparator.md
│   └── ...
├── strategy_freq_map.json      # 策略-频率映射
├── pitfalls/                   # 避坑库
│   ├── fanout_replication_high_freq.md
│   └── ...
├── ai_blindspots/              # AI 推理盲区（LEC 失败案例）
└── boundary_audits/            # 越界审计日志
```

## 八、可落地的最小实现路径

不必一次做全套，按价值/成本排序逐步建：

### V1（先解 80%）
```
✓ 阶段 0：基线建立（脚本化 Vivado 调用）
✓ 阶段 2：工具层搜索（AutoResearch 循环 + 并行）
✓ 阶段 4：知识沉淀（仅记策略-频率映射）
```
**这一版就能拿到 7-10% 的频率提升**，覆盖文章的 333→348MHz 段。

### V2（加静态预筛）
```
+ 阶段 1：静态预筛
+ LEC 闸门
+ 安全改写池
```
**这一版能多拿 3-5%**，覆盖共享比较器类优化。

### V3（加越界探索）
```
+ 阶段 3：Agent 越界
+ 完整 audit log
+ AI 盲区库
```
**这一版能突破工具层天花板**，覆盖文章的 348→360MHz 段。

### V4（知识库飞轮）
```
+ 跨项目知识复用
+ 模式抽象自动化
+ 团队级共享
```
**这一版才有"组织资产"价值**，单项目频率提升只是副产物。

## 九、风险清单与对策

| 风险 | 概率 | 后果 | 对策 |
|------|------|------|------|
| AI 提议的 RTL 改写过 LEC 但功能错 | 低 | 严重 | testbench 全回归 + 高风险改写人审 |
| AI 自评指标作弊 | 中 | 严重 | 所有指标必须 Vivado 给，禁止 AI 估算 |
| 越界次数发散 | 中 | 浪费资源 | 越界次数硬上限 + audit |
| 知识库污染（错误模式入库） | 中 | 长期累积 | 入库前过 review，可回滚 |
| 单次 PnR 成本太高 | 高 | 进度慢 | 并行 + 增量编译 + 静态预筛前置 |
| LEC 工具不支持某些改写 | 中 | 改写无法验证 | 此类改写转人审，不进自动循环 |

---

## 十、一句话总结

> **可落地的工作流 = 基线打底 + 静态预筛大杠杆 + AutoResearch 机械试错 + 触发式 Agent 越界 + 自动知识沉淀；五阶段串行，安全网 LEC + testbench + Vivado 全程兜底，AI 只提议永不裁判，失败比成功更值钱。先做 V1 拿 80%，再用 V2/V3/V4 滚动加价值。**
