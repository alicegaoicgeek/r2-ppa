## W1 · FPGA 时序优化（最完整的循环工程样板）

**来源：** 102 五阶段、99 四阶段、97 Aha、619 skill 分析

```text
        ┌─────────────┐
        │ 0 基线 PnR  │──Fmax/WNS/TB基准──┐
        └─────────────┘                   │
                                          v
        ┌─────────────┐    LEC fail    ┌──────────────┐
        │ 1 静态预筛  │──AI改写候选──►│ LEC 等价闸门 │
        │   AI+报告   │               └───┬──────┬───┘
        └─────────────┘                   │pass  │fail→避坑库
                                          v
        ┌─────────────┐              ┌─────────────┐
        │ 2 工具层搜索│◄──并行 PnR──│ AutoResearch│
        │   参数扫    │              │  读优参数   │
        └──────┬──────┘              └─────────────┘
               │ 仍不够
               v
        ┌─────────────┐    LEC+audit   ┌─────────────┐
        │ 3 越界探索  │───────────────►│ Agent 新假设│
        │   结构级    │◄──回 2 或结束──└─────────────┘
        └──────┬──────┘
               v
        ┌─────────────┐
        │ 4 知识沉淀  │──成功模式进目录 / 失败进避坑库──► 下一轮更强
        └─────────────┘
```

| 节点 | 谁做 | 闸门 |
|---|---|---|
| 0 基线 | 工具（Vivado 等） | 报告可解析 |
| 1 预筛 | AI 生成候选 | **LEC pass** 才进池 |
| 2 搜索 | 多轮 PnR 并行 | Fmax/WNS 变好则保留 |
| 3 越界 | AI + 人审高风险 | LEC + 可选人审 |
| 4 沉淀 | 写库 | 有记录可复用 |

**停止条件：** Fmax ≥ 目标，或工具层+越界层都无进展。

## W2 · AI 造 CPU / 核（研究 → 实现 → 指标闭环 → 上板）

**来源：** 516、924 合集、15

```text
[竞品参数库] → [读 TRM/开源核] → [机会地图/硬指标]
        │
        v
[架构决策] → [RTL 生成/迭代] ──► [仿真/基准]
        │                            │
        │         fail               │ pass 指标
        └──────── 改 RTL ◄───────────┘
        │
        v  (系列三)
[FPGA 集成] → [bitstream] → [上板 / OS 跑通]
        │
        v
[面积/频率/对标报告]
```

| 阶段 | 输入 | 输出 | 闸门 |
|---|---|---|---|
| 建 OS | 公开资料 | 参数表、差距图 | 目标数字可辩护 |
| 设计 | 指标+架构 | RTL | 仿真绿 |
| 性能环 | RTL | 更高 CoreMark/Dhrystone | **对标表数字** |
| 上板 | 比特流 | FreeRTOS/真实访存 | 上板行为正确 |

**本质：** 外环是「指标驱动」；内环是「改 RTL → 测 → 再改」（循环工程）。

## W3 · 一句话 / Spec → RTL → 验证 → 物理签核

**来源：** 695 / 894 AHB 总线矩阵

```text
[一句话需求]
    → [RTL + 接口判断]
    → [Scoreboard / TB]
    → [多 seed 仿真] ──fail──► 改设计/TB
    → [综合/STA/面积/fmax扫]
    → [签核报告]
```

| 节点 | 闸门 |
|---|---|
| 功能 | 测试全绿（多 seed） |
| 签核 | WNS/面积/fmax 可报告 |

**短 DAG，适合「IP 块级」交付；** 与 W1 比，少「知识沉淀」节点时只是单次作业，加上沉淀才成 Loop。

## W4 · 形式验证 / 断言

**来源：** 885–896、53 FVRuleLearner、887

### W4a · 证明会话（人+AI 共驾）

```text
[设计+目标断言]
  → [选技术：引理/bbox/抽象…]
  → [跑 formal 引擎]
  → prove? ──no──► 看反例/cone → 改断言或策略 → 再跑
  → yes → 记录手法进 skill
```

### W4b · 错题→规则（FVRuleLearner 型）

```text
[NL/任务] → [生成 SVA] → [FV 评估]
                ↑              │ fail
                └── 规则/Q-Tree 沉淀 ◄─┘
                      │
                      └── 下次生成时注入 prompt
```

**闸门：** 引擎证明结果，不是「AI 觉得对」。

## W6 · 覆盖率收敛（验证最后一公里）

**来源：** 891

```text
[configure 模型角色]
  → [读 cov 报告 / hole]
  → classifier 归类 hole
  → case_writer 写激励
  → sim
  → reviewer 看是否真盖住
  → hole 还在? ──yes──► 再写 case
  → no → 收敛报告
```

**三角色：** classifier / case_writer / reviewer（写≠查）。  
**闸门：** 覆盖率数字 + 可选人审关键 hole。

## 跨案例的「公共骨架」（所有 DAG 的母版）

几乎所有可交付案例都落在这张母图上：

```text
         目标(数字/断言/覆盖率)
                 │
                 v
    ┌──── 读上下文 / Skill / 状态 ────┐
    │                                  │
    v                                  │
  执行(改RTL/参数/TB/断言)               │
    │                                  │
    v                                  │
  硬闸门(LEC/sim/formal/lint/bench/PnR) │
    │                                  │
    ├─ fail ──► 记原因 ──► 再执行 ─────┘
    │
    └─ pass ──► 沉淀(规则/库/报告) ──► 人签核
```

| 母版节点 | caseall 里的名字 |
|---|---|
| 目标 | Fmax、CoreMark、prove、cov%、签核 |
| 执行 | AI / AutoResearch / Agent |
| 硬闸门 | LEC、Vivado、pytest/sim、Formal、Superlint |
| 沉淀 | 安全池、避坑库、Q-Tree 规则、Skill |
| 人 | 高风险改写、架构裸读、最终签核 |

**这就是你要的「每个工作的 DAG」的统一语法。**  
差别只在：**目标是什么、闸门用哪条命令、沉淀写回哪里。**

## 建议：先做哪几张 DAG（交付优先级）

| 优先级 | DAG | 理由 |
|---|---|---|
| P0 | **W1 时序优化** | 五阶段写得最完整，循环工程最像教科书 |
| P0 | **W3 RTL→签核** | 路径短，客户一眼懂 |
| P1 | **W6 覆盖率** | 验证刚需，角色分离清晰 |
| P1 | **W4 形式** | 差异化，闸门硬 |
| P2 | **W2 造核** | 故事强，但周期长，适合标杆不是日课 |
| P2 | **W8 永续优化** | 与 W1 同构，换 bench |

## 和「工具」的关系（只一句）

DAG 是 **作业怎么转**；  
Claude Code / omp / Grok 只是 **执行节点上的手**。  
闸门节点应尽量是 **Vivado / LEC / sim / formal / lint**，与用哪家对话模型无关。
