# AutoResearch × FPGA 时序优化：可落地工作流

## 一、解决什么问题

**表面问题：** 给定一个时序未收敛的 FPGA 设计（如 324 MHz），推到目标频率（如 360 MHz）。

**实质问题：** FPGA 时序优化是经验密集型、试错成本极高、知识难以复用的领域。这个工具把"试错过程"变成"知识资产"——单个项目的频率提升是副产品，**可复用的知识库才是核心产出**。

---

## 二、四阶段工作流

```
工程师输入
(RTL + 目标频率 + 边界)
      │
      ▼
┌─────────────────────────────────────────┐
│ Phase 0: 基线 + 静态预筛                │
│ 跑一次 PnR 建立基线                      │
│ AI 扫 RTL 找等价改写 → LEC 验证 → 安全池│
│ 时间：基线几小时 + 预筛分钟级            │
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│ Phase 1: 工具层循环（AutoResearch）      │
│ Vivado 选项轮转：布局/扇出/DSP/SRL/...  │
│ 每轮：改参数 → 跑 PnR → 读 WNS → 选留弃 │
│ 并行跑 N 个候选，对抗单次成本高          │
│ 推到工具层天花板                         │
└──────────────────┬──────────────────────┘
                   │
            ┌──────┴──────┐
            ▼             ▼
       达到目标      撞天花板
            │             │
            ▼             ▼
         输出结果   ┌──────────────────┐
                    │ Phase 2: Agent 越界│
                    │ 读 RTL 找新改写    │
                    │ 读 log 判断趋势    │
                    │ 新候选回 LEC → 回  │
                    │ 到 Phase 1 验证    │
                    └──────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────┐
│ Phase 3: 知识沉淀（自动）               │
│ 成功 → 抽象为模式，写入工具目录         │
│ 失败 → 注释写回源码，记入避坑库         │
│ LEC 失败 → 记录 AI 推理盲区            │
└─────────────────────────────────────────┘
```

## 三、每阶段的输入、执行、输出

### Phase 0：基线 + 静态预筛

```
输入：
  - RTL 源码（SystemVerilog）
  - 约束文件（XDC）
  - 目标频率（如 360 MHz）
  - 边界条件（不许多耗资源、testbench 必须过）

执行：
  1. 跑一次完整综合+布局+布线 → 建立基线
  2. 读时序报告，识别关键路径瓶颈模式
  3. AI 静态扫 RTL，寻找等价改写候选：
     - 公共子表达式复用
     - 冗余比较器/加法器
     - "裁剪后再判零"等代数模式
  4. 每个候选过 LEC（形式化等价检查）
  5. 通过的写入"安全改写池"，失败的记避坑库

输出：
  - 当前 Fmax / WNS 基线
  - 安全改写池（LEC 验证过的 RTL diff）
  - 关键路径报告（路径清单 + 瓶颈分类）

决策点：
  - Fmax >= 目标？ → 直接结束
  - 安全池为空？ → 跳过改写，进 Phase 1
```

### Phase 1：工具层循环（AutoResearch）

```
输入：
  - 当前基线设计（可能已应用安全改写）
  - 安全改写池
  - Vivado 选项目录（布局策略、phys_opt、扇出、DSP 映射、SRL 推断等）
  - 目标 Fmax

循环：
  while Fmax < 目标 and 未收敛:
    1. 从选项目录选出 N 个候选方案（并行）
    2. 每个候选：
       a. 修改约束/选项文件（不改 RTL）
       b. 跑 PnR
       c. 读 WNS / Fmax / 资源
       d. 优于当前 best？ → keep，成为新基线
       e. 更差？ → discard，失败原因写回知识库
    3. 连续 M 轮无改善？ → 触发 Phase 2

输出：
  - 最优 Fmax 及对应配置
  - 完整迭代日志（每轮的选项 + WNS + 决策）
  - 失败记录库

决策点：
  - Fmax >= 目标？ → 结束，进 Phase 3
  - 连续 M 轮无改善？ → 进入 Phase 2 越界探索
```

### Phase 2：Agent 越界探索（触发式）

```
触发条件：Phase 1 推到天花板但未达目标

输入：
  - Phase 1 结果（当前 Fmax、迭代日志、穷尽的选项列表）
  - RTL 源码
  - PnR 迭代日志

执行：
  1. 读迭代日志：
     - 判断 WNS 曲线是"游荡"还是"发散"
     - 游荡且接近 0？ → 建议增加迭代上限
  2. 重读 RTL 源码：
     - 找 Phase 0 漏掉的改写机会
     - 跨抽象层分析（如协议层、状态机层）
  3. 新改写候选 → 过 LEC → 通过则进安全池
  4. 修改硬编码参数（如迭代上限）
  5. 回到 Phase 1 验证

输出：
  - 新 RTL 改写（LEC 验证过）
  - 新参数调整
  - 回到 Phase 1 或确认已达天花板

风险：高。所有输出必须回 LEC + testbench 才能进循环。
```

### Phase 3：知识沉淀（自动，每轮结束后）

```
输入：
  - 所有成功方案（WNS 改善的）
  - 所有失败方案（WNS 变差的）
  - LEC 失败记录

执行：
  1. 成功模式 → 抽象为目录条目
     - 命名模式（如"共享比较器复用"）
     - 标注适用条件（频率范围、瓶颈类型）
  2. 失败方案 → 注释写回源码
     - "在此条件下试过 XXX，因为 YYY 导致更慢"
     - 下次人或 AI 走到同一路径不再重复
  3. LEC 失败 → 记录"AI 推理盲区"案例
     - "AI 以为 A≡B 但 LEC 证明不等价"

输出：
  - 更新后的瓶颈分类目录
  - 更新的避坑表（源码注释 + 知识库）
  - 更新的策略-频率映射表

价值：下次项目遇到同类情况，Phase 0 起点更高。
```

## 四、RTL 改写安全规则

**核心原则：RTL 改写不进 Phase 1 循环，只在 Phase 0 和 Phase 2 发生，且必须过 LEC。**

| 改动类型 | 能在循环里改？ | 安全要求 |
|---------|:---:|---------|
| Vivado 选项（布局/phys_opt/扇出属性） | ✅ | Vivado 保证等价，改错回滚即可 |
| 纯组合逻辑代数化简 | ⚠️ LEC 后 | LEC + testbench 回归 |
| 流水线插一拍（改语义） | ❌ | 必须人审，下游时序假设变了 |
| 状态机重编码 | ❌ | LEC + 人审 |
| Reset 逻辑改动 | ❌ 永远不行 | 隐藏最深的 bug 来源 |
| 时钟门控 / CDC | ❌ 永远不行 | LEC 都不一定能查出 |

**三道防线（按强度排序）：**

1. **testbench 回归（弱）** — 快速排除明显错误，但只覆盖测过的输入
2. **形式化等价检查 LEC（强）** — Vivado LEC / Yosys eqy，数学证明所有输入下输出相同，**工业硬门槛**
3. **人类 review（最终）** — 动协议、状态机、reset、时钟域的必须人审

**关键纪律：指标必须由 Vivado 给，不能 AI 给。**
- WNS、Fmax、资源用量全部来自 Vivado 报告
- 功能正确性由 testbench / LEC 保证
- AI 不能自己估算时序或判断"等价改写"是否正确

## 五、失败模式与应对

| 坑 | 表现 | 应对 |
|---|------|------|
| **迭代曲线非单调** | 第 5 轮 WNS 突然变最差，回滚就前功尽弃 | 看趋势不是看点值，游荡不等于发散，给足轮次 |
| **局部优化挤散全局** | 扇出复制局部快了，全局反而慢 166ps | 每次只看全局 WNS，不看局部报告 |
| **DSP 映射不一定更快** | 专用硬件离数据源太远，连线绕远慢 675ps | 纳入候选让数据说话，不预设"专用=快" |
| **默认值不等于最优** | Vivado 默认 5 轮物理优化停在最差态 | 把默认值当基线不是当边界 |
| **LEC 失败** | AI 自以为等价但实际不等价 | 记入"推理盲区"，比试错失败更有价值 |

## 六、IO 速查

| 层级 | 输入 | 输出 | 时间 | 风险 |
|------|------|------|------|------|
| Phase 0 基线 | RTL + XDC + 目标 | 当前 Fmax + 关键路径报告 | 几小时 | 无 |
| Phase 0 预筛 | RTL + 关键路径 | 安全改写池（LEC 验证） | 分钟~小时 | 低 |
| Phase 1 循环 | 安全池 + 选项目录 | 最优配置 + 迭代日志 | 小时×N | 低（可回滚） |
| Phase 2 越界 | 卡住的状态 + 全局上下文 | 新假设 + 新候选 | 分钟 | **高（必须 LEC）** |
| Phase 3 沉淀 | 全部日志 | 知识库更新 | 分钟 | 无 |

## 七、与原始 AutoResearch 的差异

| 维度 | AutoResearch (ML) | FPGA 时序优化版 |
|------|-------------------|----------------|
| 可改什么 | train.py | Phase 0/2：RTL（需 LEC）；Phase 1：Vivado 选项 |
| 不可改什么 | prepare.py + eval.py | 功能规范、testbench 不能动 |
| 单次迭代 | 5 分钟 | 30 分钟 ~ 数小时 |
| 策略 | 单线程串行 | 并行 N 候选，对抗单次成本 |
| 指标 | Loss | WNS（Vivado 给，不可篡改） |
| 安全网 | 无（Loss 就是真理） | LEC + testbench + Vivado 报告 |
| 知识复用 | 有（价值中等） | 极高（每次试错贵 100 倍） |
| 越界 | 不允许 | Phase 2 允许，但必须回 LEC 验证 |

---

## 八、一句话

> **Phase 0 静态预筛找安全改写 → Phase 1 AutoResearch 循环做工具层穷举 → 撞天花板进 Phase 2 Agent 越界 → Phase 3 知识沉淀"发现一次下次免费"。全程 LEC 护 RTL、Vivado 护指标、testbench 护功能，AI 只负责在安全边界内最大化搜索效率。**
