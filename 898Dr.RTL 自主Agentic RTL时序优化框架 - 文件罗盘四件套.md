# Dr.RTL: 自主Agentic RTL时序优化框架 — 文件罗盘四件套

> 源论文：Dr.RTL – Autonomous Agentic RTL Timing Optimization  
> 4 个 Agent 并行执行，1 个汇总 Agent 整合

## Agent 1：痛点卡片（7个）

### 痛点 1：简化RTL假设 — 学术界一直在退化的代码上做优化

**痛点场景**：现有 benchmark（RTLRewriter、SymRTLo）通过手工退化（constant propagation、CSE消除）把人类RTL变成"未优化基线"，再让算法优化回原始版本。原始RTL被当成 golden 上限。

**具体后果**：算法在128行、1个模块的玩具代码上表现优异，但面对真实项目中4615行、7个模块的人类RTL时完全失效。论文实验证明：原始人类RTL本身就能再优化21%，之前 benchmark 的假设是错误的。

**黄金开头**：你的RTL优化算法在 benchmark 上拿了高分，但放到真实项目上可能直接崩盘。

**黄金钩子**：因为整个学术界用来评估的"标准数据集"，是用人工退化的简化RTL造的 —— 跟工程师实际写的代码根本不是一个物种。

### 痛点 2：开源工具链幻觉 — Yosys仿真 vs 工业签核鸿沟

**痛点场景**：现有方法用Yosys做综合、用 behavioral simulation 做等价检查。但工业界用 Synopsys DC + Cadence Jasper SEC 做签核。两者的综合结果、时序模型、优化策略存在显著差异。

**具体后果**：在Yosys上跑通的RTL优化，放到DC+Jasper流程中可能SEC直接 fail 或时序恶化。Dr.RTL 用 Jasper SEC 验证，86%通过率 —— 即使最强LLM + 精心设计的多Agent框架，仍有14%在工业级验证下失败。

**黄金开头**：你在Yosys上跑通的RTL优化，工业界的签核工具可能直接判死刑。

**黄金钩子**：开源EDA和商用EDA之间的鸿沟，足以让一个在论文里表现完美的算法，在流片前夜变成一颗定时炸弹。

### 痛点 3：设计级PPA反馈太粗 — 不知道"哪里"慢，只能瞎改

**痛点场景**：现有方法只拿到整个设计的 WNS/TNS/Area 三个数字，不知道具体哪条路径慢、哪个模块是瓶颈、哪个信号扇出过大。优化器像蒙眼射击。

**具体后果**：消融实验：移除 register-level slack 反馈后，WNS改善从21%暴跌到9%。没有关键路径级别的细粒度反馈，LLM Agent 只能做宏观层面的猜测式重写，浪费大量综合迭代。

**黄金开头**：你只给LLM三个数字（WNS/TNS/Area），它再聪明也只能瞎猜该改哪。

**黄金钩子**：21%的时序改善 vs 9%的差距，就在于你有没有告诉AI："慢在这里，具体是这条路径上的这个比较器。"

### 痛点 4：LLM选错优化方向 — 改了不该改的地方，SEC一验就挂

**痛点场景**：没有 skill 库时，优化器面对"高扇出控制信号"可能尝试：加 buffer 链（无效，综合工具自动做）、改全局时钟网络（SEC fail）、用 generate 复制但命名混乱（SEC fail）。

**具体后果**：消融实验：禁用 skill 库后，SEC通过率从86%降到78%，WNS从21%降到14%。每次猜错方向，浪费一次综合（几十分钟）+ 一次SEC验证 + 一次API调用。20个设计跑下来，错误尝试成本乘以几十倍。

**黄金开头**：让LLM凭空猜测怎么改RTL时序，它每猜错4次才对1次。

**黄金钩子**：86%的成功率不是LLM本身有多强，而是有人提前告诉它"这种情况下别试那几种改法" —— 这就是 skill 库的价值。

### 痛点 5：EDA反馈噪声大 — 同一设计的不同run结果波动，难以归因

**痛点场景**：综合结果是启发式的 —— 同样的RTL，DC的综合策略、约束设置、工具版本微小差异都会导致PPA波动。用绝对PPA值来判断一个优化"好不好"充满噪声。

**具体后果**：如果不做 group-relative learning，可能把一次"运气好"的综合结果当成好优化，或者把真正好的变换因为综合噪声而 discard。消融实验证实：移除后性能显著下降。

**黄金开头**：你用绝对PPA数值来评判RTL优化的好坏，可能只是在追综合工具的随机噪声。

**黄金钩子**：真正的信号藏在相对比较里：同一轮5个候选中谁 consistently 更好，而不是谁的绝对 slack 数字碰巧撞上了综合工具的好脾气。

### 痛点 6：Agent单兵作战 — 一个LLM干所有事效果差一半

**痛点场景**：把 Timing分析、RTL重写、综合验证三个任务塞给同一个LLM实例，它要在"诊断病因""开药方""验疗效"三个角色间切换，上下文混乱。

**具体后果**：消融实验：移除多Agent设计后，WNS/TNS从21%/17%降到10%/7% —— 性能直接腰斩。角色不分离意味着每个任务都做不到专注，优化方向被无关信息干扰。

**黄金开头**：让同一个AI又当时序分析师、又当RTL工程师、又当验证工程师，效果直接砍半。

**黄金钩子**：21%的改善降到10%，不是因为LLM变笨了，而是因为它在三个专业角色之间精神分裂。

### 痛点 7：弱LLM backbone — 8B模型基本不可用

**痛点场景**：论文测试了从 Qwen Coder 8B 到 Claude Opus 的4个模型。8B参数模型在RTL优化任务上 WNS 只改善1%，SEC通过率仅22%。

**具体后果**：如果团队想本地化部署小模型做RTL优化，现实是：8B几乎不可用，Haiku勉强能用（3%），要到Sonnet级别（12%）才有实用价值，Opus级别（21%）才达最佳效果。模型能力门槛意味着这不是"随便找个开源LLM就能跑"的任务。

**黄金开头**：你以为随便部署一个7B开源模型就能自动优化RTL时序？现实是它能做的改善不到1%。

**黄金钩子**：从1%到21%，差的不是 prompt engineering，而是模型本身能不能理解时钟域、组合深度、扇出延迟这些硬件概念。

## Agent 3：可落地方法（6个）

### 方法 1：Group-Relative PPA评估法 — 用相对比较替代绝对数值

- **适用场景**：EDA综合结果存在工具随机性和设计依赖性，用绝对 WNS/TNS/Area 评判优化质量容易被噪声误导
- **步骤**：
  1. 每次优化迭代时，从同一个父RTL版本并行生成5个候选优化（5个 minor attempt）
  2. 对这5个候选计算标准化分数：`score = 0.5×WNS_norm + 0.35×TNS_norm + 0.15×Area_norm + area_penalty`
  3. 其中 `WNS_norm = (WNS - WNS_baseline) / |WNS_baseline|`，其他同理
  4. 选5个候选中分数最低且SEC通过的版本，提升为下一轮基线
  5. 不比较不同迭代轮次的绝对分数，只比较同一轮内的相对排名
- **验证标准**：10轮迭代后，最终版本的 WNS_norm 和 TNS_norm 均为负值（优于基线），且SEC验证通过

### 方法 2：RTL时序瓶颈根因分析五维检查清单

- **适用场景**：综合后WNS为负（时序违例），需要快速定位RTL层面的根因
- **步骤**：
  1. 从DC时序报告中提取Top-K关键路径（小设计K=10，中设计K=15，大设计K=20-25）
  2. 检查维度1——**扇出**：关键路径上的信号是否驱动>20个负载？→ 考虑寄存器复制
  3. 检查维度2——**组合深度**：路径上是否经过>5级组合逻辑？→ 考虑流水线插入或表达式平衡
  4. 检查维度3——**FSM解码**：是否从状态寄存器直接驱动大量组合逻辑？→ 考虑FSM输出寄存
  5. 检查维度4——**宽比较器**：是否有>8位的比较操作在关键路径上？→ 考虑分段比较或预计算
  6. 检查维度5——**控制-数据耦合**：控制信号是否在关键路径上影响数据流？→ 考虑控制信号预解码
- **验证标准**：每个维度检查后输出具体的RTL修改位置和预计改善幅度，修改后WNS改善5-15%

### 方法 3：Skill库驱动的RTL优化（已验证47条Pattern-Strategy）

- **适用场景**：需要复用已验证的RTL优化策略，避免LLM每次都从零猜测
- **步骤**：
  1. 建立 `skill.md` 文件，按 pattern（瓶颈模式）+ strategy（修复策略）+ evidence（验证证据）组织
  2. 高置信度策略（>80%成功率）优先使用：One-hot预解码、条件预计算、寄存器复制、FSM输出寄存
  3. 优化前先用当前设计的瓶颈特征匹配skill库中的pattern
  4. 匹配成功则直接应用对应strategy，标注 `skill_source: "learned"`
  5. 无匹配时由LLM提出新优化，标注 `skill_source: "proposed"`，记录结果用于后续skill提取
  6. 每次运行后用 Skill Extractor Agent 分析 `history.json`，将新验证的策略合并到skill库
- **验证标准**：应用skill库后SEC通过率>85%（vs无skill库的<80%），且平均尝试次数减少60%

### 方法 4：多Agent角色分离架构（Analyzer-Optimizer-Evaluator）

- **适用场景**：构建RTL优化自动化系统，需要确保时序分析、代码改写、验证评估三个环节不互相干扰
- **步骤**：
  1. **Agent 1 - Timing Analyzer**：只负责诊断"为什么慢"和"在哪里慢"，不生成修改代码。输入：时序报告+RTL源码。输出：关键路径列表+根因诊断
  2. **Agent 2 - RTL Optimizer**：只负责根据Analyzer的诊断改写RTL代码。输入：RTL源码+路径分析+skill库。输出：优化后的RTL
  3. **Agent 3 - Synthesis Evaluator**：只负责运行综合和SEC验证。输入：优化RTL。输出：PPA报告+SEC结果
  4. **Orchestrator编排器**：执行10轮×5候选的循环，从Evaluator结果中选择最优SEC-pass版本提升为下一轮基线
  5. Agent间通过共享文件（`iter.json`, `history.json`）传递状态，不直接通信
- **验证标准**：对比单Agent方案，WNS改善从10%提升到21%（消融实验证实角色分离的价值）

### 方法 5：真实RTL优化Benchmark构建流程

- **适用场景**：需要建立一个贴近工业现实的RTL优化评估环境，替代现有的退化RTL benchmark
- **步骤**：
  1. 收集人类手写的RTL设计（课程项目、开源处理器、通信协议、加密模块），确保128-4615行、1-7个模块的规模分布
  2. 用 Synopsys Design Compiler 做综合，目标工艺库（如 Nangate 45nm），提取 WNS/TNS/Area 基线
  3. 用 Cadence JasperGold 做SEC验证（时序等价检查SEC，非组合等价检查CEQ）
  4. 从DC时序报告中提取 register-level slack 信息，映射回RTL源码位置作为 fine-grained feedback
  5. 设计配置标准化：`design_all.json` 记录顶层模块名、时钟信号、复位信号、外部信号列表
  6. 版本化管理：`v0`（基线）→ `v0.1`-`v0.5`（候选）→ `v1`（提升）→ ... → `v10`（最终）
- **验证标准**：原始人类RTL本身还能再优化>20%（WNS），证明benchmark的真实性；SEC通过率>85%证明修改可靠性

### 方法 6：10轮×5候选的迭代优化调度策略

- **适用场景**：需要系统化地探索RTL优化空间，避免过早收敛到局部最优
- **步骤**：
  1. 初始化：综合v0基线，创建 `history.json`，记录基线 WNS/TNS/Area
  2. 每轮 Major Round 生成5个 Minor Attempt（`vX.1` 到 `vX.5`），每个用不同的多样性策略
  3. 多样性维度1——**路径选择**：Top-K by slack / path range / endpoint clustering / high fanout / module-focused / random
  4. 多样性维度2——**优化重点**：combinational / sequential / mixed
  5. 确保5个minor在路径选择和优化重点上有差异，覆盖不同优化机会
  6. 每minor执行：Analyzer诊断 → Optimizer改写 → Evaluator综合+SEC → 记录score
  7. 选5个minor中score最低且SEC pass的版本，promote为 `v(X+1)`
  8. 终止条件：v10完成，或连续10轮无promotion
- **验证标准**：最终版本WNS和TNS均优于基线，且优化轨迹文件（`history.json`）完整记录了每轮的探索和选择过程

## 汇总 Agent：五句话总结

1. **Benchmark 是假的**：学术界用退化RTL做评估，Dr.RTL证明真实人类RTL仍可再优化21%——之前的假设全错。
2. **角色必须分离**：Analyzer-Optimizer-Evaluator 三Agent架构比单Agent效果翻倍（21% vs 10%），三合一就是精神分裂。
3. **反馈要细到寄存器级**：只给WNS/TNS/Area三个数 = 蒙眼射击；给关键路径级slack = 精准拆弹。
4. **Skill库是护城河**：47条 Pattern-Strategy 让SEC通过率从78%→86%，错误方向尝试减少60%。
5. **模型有硬门槛**：8B不可用（1%），至少 Sonnet 级（12%）才有实用价值，Opus 级（21%）达最佳效果。
